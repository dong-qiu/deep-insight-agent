# Spec: 按源贡献 / ROI（ADR-0008 决定⑦ / 切片4）

> 设计文档 v2（多 Agent 评审后**降级**）。落 ADR-0008 决定⑦的**诉求**（揪常年 0 贡献的源），但放弃精确成本摊派。
> v2 裁决：**砍成「源健康看板的一个分子列」并入切片3**——不建表、不摊分母、不告警。精确成本归因整体推后。

## 背景与目标

源管理漏了最贵一维：每源占用 analyzer 预算 vs 换回的被采纳洞察。决策诉求是**二元的**：哪些源常年 0 贡献、值不值得继续每轮花钱分析。评审结论：**这个二元判断只需「分子信号」即可回答，精确分母（每源分析成本）既做不准也不必做**。

## 为什么放弃精确成本摊派（评审🔴，留痕）

v1 方案「新增 batch_source_input 记每源 char 数、按 char 占比摊 analyze run.cost」被否，三条硬伤：

1. **捕获点接错代码**：v1 写「analyzer.ts `selectForAnalyze` 之后捕获」——但 `selectForAnalyze`（`analyzer.ts:277`）是单条 body 的截断分窗、**不是选 item**；`analyze()`（`analyzer.ts:490`）是**无 DB 句柄的纯函数**、无法落库。唯一能同时拿 db+items+batch.id 的是 `runAnalysis`（`pipeline.ts:25`）。又一例「设计没接上代码」。
2. **char 占比摊派统计无效**：① 正文硬截到 `ANALYZE_BODY_CHARS=10000`（富源被压平、占比失真）；② 输出 token（约输入 5×）与单源输入 char 无因果；③ system prompt 走 prompt cache 是**源无关固定成本**。`run.cost` 只存 `{tokens, amount}` 总量（`types.ts:163`），连「只摊非缓存输入」都做不到。
3. **二元决策不需要分母精度**：揪「0 贡献源」只看分子是不是 0，分母多准都不改变结论。

> 反向洞察（留作未来）：真要摊成本，**validate 比 analyze 更可按源归因**（validate 按被引 item 发生、走 citation→source 同一条链）；report-gen 零 LLM（`pipeline.ts:60`）。但**当前连 validate 摊派也先不做**。

## 用户故事

- As 运维, 我想在源健康看板看到**每源近 N 窗被多少份已上报报告的洞察引用**，一眼看出常年 0 贡献的源。

## 输入 / 输出契约

| 项 | 说明 |
|---|---|
| 输入 | `report`（已上报）→ insight_ids → `insight` → `citation.content_item_id` → `content_item.source_id`（**全部已落库，零新表**） |
| 输出 | 源健康看板新增「贡献」列：`{source_id, 近 N 窗被引洞察数(计数), 时间窗}`（与切片3 健康表同页） |
| 触发 | 看板**读时聚合**（无新捕获、无 job、无 schema 改动） |

## 行为规约

1. **分子 = 计数（非均摊）**（评审裁决）：对每份已上报 `report` 的 insight_ids → 每条 insight 的 citations → distinct source_id 集 → 每被引源「贡献被采纳洞察数」+1。multi-source 洞察对每个被引源**各 +1**（不 1/n 均摊——均摊会压低稀疏但常被多源佐证的权威源 OWASP，与决定⑦「成本低≠不重要」冲突；且二元「是否 0 贡献」判断用不上均摊的横比优势）。
2. **读时聚合**：看板查询走现有 citation→content_item.source_id 链，按 source × 时间窗聚合，零新表/零捕获/零 schema 改。
3. **只展示、不告警**（评审升为硬 AC）：初期样本/窗口不足，只展示原始计数、**不算单位成本、不标红、不告警**，避免高方差早期数据误判。攒 ≥4–6 周后若要告警，用分位数（最差 10%）而非绝对阈值。
4. **与切片3 零产出看门狗去重**：零产出（采集层 inserted=0，要快）与本列（分析价值层 被采纳=0，慢、人工）严重重叠。分工：零产出只管采集层；本列只展示不告警 → 天然去重，一个死源不双告警。
5. **口径警示（写进看板文案）**：「未被引」≠「无价值」——analyzer 宁短勿拼只挂部分 citation（`analyzer.ts:31`）。文案只陈述「近 N 窗未被任何上报洞察引用」，**禁止断言「该源无价值」**，判断交人。

## 验收标准 (AC)

- [ ] AC1: 看板「贡献」列正确显示每源近 N 窗被已上报洞察引用的 distinct 洞察数（计数口径）。
- [ ] AC2: multi-source 洞察对每个被引源各计 1。
- [ ] AC3: 纯读时聚合——**零新表、零 schema 改动、零捕获代码**。
- [ ] AC4: **只展示不告警**；与切片3 零产出看门狗无重复告警。
- [ ] AC5: 文案不断言「无价值」，只陈述「未被引用」。

## 非功能要求

- 性能：读时聚合，看板查询级；数据量大时可加索引（citation 已有 PK，content_item 已有 idx_content_source）。
- 成本：零 LLM、零新写入。
- 可观测性：进 admin 源健康看板（与切片3 同页）。

## 依赖与影响范围

- **schema**：无（纯读）。
- **改**：`runtime/run-stats.ts` 或新 `source-contribution.ts`（聚合纯函数 + 单测）、admin 源健康看板（加「贡献」列）。
- **依赖**：与切片3 同页出（并入其看板改动）；不依赖任何新捕获。
- **历史数据**：现有 citation/report 即可回溯，**无需等新数据积累**（这是降级后的额外好处——v1 的 batch_source_input 只能实施后才有数据）。

## 推后项（待分子信号被证明不足再做）

1. **精确成本摊派**：若日后真要看「每源花了多少钱」，**优先做 validate 按源归因**（走 citation 同链），不做 analyze char 摊派。需届时新增成本捕获 + 同事务落库。
2. **低 ROI 告警**：攒足样本后用分位数阈值。
3. **均摊口径辅列**：如需横比可加 1/n 均摊列，但主口径保持计数。

## 开放问题（已裁决，留痕）

- 做精确 vs MVP vs 推后 → **降级为分子计数看板列**，精确成本摊派推后。
- multi-source 口径 → **计数（各 +1）**，非均摊。
- 是否告警 → **初期只展示**。
- 分子/分母数据 → 分子用现有 citation 链（零捕获）；分母不做。
