## 一、硬缺口：MVP 范围里写了、但没做或只做一半

### 1.1 账号体系 —— 只有单管理员，注册与数据隔离缺失 → ✅ 已决策推后（2026-06-07）

- **承诺（原）**：`product-definition.md` 旧版「最小账号体系（注册 / 登录 / 数据隔离）」；安全设计 line 324「配置 / 数据 / 报告按账号严格隔离」。
- **实际**（`src/auth.ts:15-26`）：单管理员，凭据来自环境变量 `ADMIN_EMAIL/ADMIN_PASSWORD`，JWT session；**无注册流程**。
- **根因层级**：reports / topics / sources 等表**没有 userId 维度**——`src/app/page.tsx`、`src/app/settings/page.tsx` 直接取全量数据。"按账号隔离"在数据模型层就不存在，不是 UI 未接。
- **决策（2026-06-07）**：**MVP 口径收敛为单用户**——单管理员登录即满足，自助注册与多账号数据隔离推后。已落到 `product-definition.md` 六处（v0.7）：①MVP 包含 账号行重写；②MVP 暂不包含 新增"自助注册 / 多账号数据隔离"行；③账号与权限节"按账号隔离"标为目标态 + 加 MVP 口径说明；④安全设计「身份与隔离」行隔离要求标为目标态/MVP 推后（保留鉴权/会话/管理员独立鉴权在范围内）；⑤安全设计「传输与存储」行"删除账号时清除关联数据"标为目标态/MVP 推后；⑥安全设计「合规与版权」行"账号注销与数据删除"标为目标态/MVP 推后。**口径不一致已闭合**——不再是缺口，而是显式推后项。

### 1.2 成本配额 / 上限告警 / 限流 —— 有度量，无管控 → ✅ 已闭合（2026-06-13）

- **承诺**：`product-definition.md:288-291`「可配成本上限（月/日/单次）；触顶告警 + 限流（暂停非关键任务）；成本可视化」。
- ~~**实际**：度量侧已做；可配置上限 + 触顶自动限流/暂停采集未落地。~~
- **闭合（✅ 2026-06-13，见 ADR-0003）**：新增 `src/lib/runtime/cost-guard.ts`（`loadBudgetLimits` / `evaluateBudget` / `getBudgetStatus`）+ `repos.sumRunCostSince`（`SUM(json_extract(cost,'$.amount'))`，日/月 UTC 窗）。三档判定（ok/alert/exceeded），env 配置 `COST_LIMIT_DAILY` / `COST_LIMIT_MONTHLY`（USD）/ `COST_ALERT_PCT`（默认 80）。**自动管线**（`runScheduledPipeline`）每 topic 前查预算，`exceeded` 硬熔断跳过剩余 topic + 告警；**手动操作**（深挖 / 追问）放行但提示（用户主动意图保留应急能力，决策见 ADR-0003）。告警复用 `notifyBudget`（多渠道层）、每 cron 进程去重。admin 顶部「成本预算」卡片显示今日 / 本月用量 + 进度条 + 状态徽标。**未配上限 = 不限 = 零回归**。+30 测试（cost-guard 各档 + 脏输入 + 日/月窗、sumRunCostSince、notifyBudget），typecheck / 410 测试 / build 全绿。见 `operations.md §14`。
- **遗留**：单次（per-call）上限未做（架构原 `quota` 表 per-topic/model 随多账号推后）；真实成本账单定稿仍被直连 key 外部阻塞（见 §1.3）。**待真机端到端验证**（配 `COST_LIMIT_DAILY` 触发一轮 cron 看熔断 + 告警）。

### 1.3 真实成本账单未定稿（外部依赖阻塞）🟡

- roadmap M4 附条件①：直连 `sk-ant-` key 不可得 → Sonnet 降本测不了、Anthropic 列表价干净测算做不了，validator 锁在 Opus relay 上是成本大头。
- **判定**：已知、被外部依赖阻塞，非遗漏。

### 1.4 深挖（deep_dive）窗口与报告结构未达 spec 🟡（2026-06-14 dogfood 核查发现）

> §二 总表把"报告"判为 ✅ 达成；逐项核实 deep_dive 后，有两处 spec-vs-实现偏离，应降级为"主体达成、deep_dive 打折"。

- **窗口 90 天 → 实测默认 14 天**：spec `report-generation.md:27`「时间窗 = 该 `Topic` 最近 90 天，可配置」；实现 `scheduler.ts:255` `windowHours` 默认 **336h = 14 天**（`DEEP_DIVE_WINDOW_HOURS` 可配）。可配性在，但默认值差 ~6.4×——"深度"综述实际只覆盖两周，与"主题深挖"的产品语义有落差。**根因可能是成本**（90 天 × 多源易超 25 条 item 上限 / 超单轮成本），但代码与注释未写明取舍，属隐式偏离。
- **完整六段 → 实测两节**：spec `report-generation.md:31` + `product-definition.md:72`「deep_dive 为完整六段：TL;DR + 关键发现 + 趋势分析 + 对比表 + 时间线 + 引用清单」；实现 `report-gen.ts` `renderMarkdown` deep 路径仅按重要性分 **「重点关注 / 其他动态」两节**（每条洞察展开来源数 / 多源印证 / 置信度 + 行内引用），**无 TL;DR / 对比表 / 时间线段**。`product-definition.md:208,231`「TL;DR 上浮第一优先 · 报告是产品心脏」的可读性主张未在 deep_dive 版式落地。
- **判定**：功能链路闭环成立（能产出可溯源的 deep_dive 报告），但**版式与覆盖窗口比 spec 设想简化**。非硬缺口、不阻塞，但与"差异化护城河 = 可读的结构化综述"（`roadmap.md:74`）直接相关，建议列为报告版式后续打磨项：①窗口默认值与成本权衡显式化（写进 decisions）；②deep_dive 套用六段模板（至少补 TL;DR + 时间线）。
