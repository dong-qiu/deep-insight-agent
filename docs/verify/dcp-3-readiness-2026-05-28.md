## 2. 核心论点：可达性 100% 是「发布层」红线，已构造即成立

`eval-criteria.md`（第 42/49 行）把「引用可达性通过率 100%」定义为**可溯源底线 / 发布门**红线。本系统在**发布层**对它做**确定性强制**，而非概率达标：

1. `validator.checkReachability`（`validator.ts:25`，纯函数无 LLM）：归一空白后判 `quote` 是否**逐字出现**在被引 `ContentItem.body`——不在即 `fail`。
2. `verdictFor`（`validator.ts:38`）：reachability=fail → verdict=`blocked`（短路，一致性不再评）。
3. `report-gen.selectInsights`（`report-gen.ts:17`）：**白名单**——只保留 verdict 为 `pass`/`flagged` 的引用下标，`blocked` 与「未校验」一律剔除（L31）；洞察若无幸存引用则整条排除。
4. 渲染只遍历 `citationIndices`（md L100-104 / html L151-156）。

**结论：进入发布报告的每一条引用，必然 verdict=pass/flagged → 逐字可达。发布层可达性 = 100%，by construction。**

**那「89%」是什么？** 是 analyzer **产出层 yield**——它产出的引用中 89% 逐字可达，另 11%（长/口语化脏内容上的真改写）被上述闸门**正确挡下**（要么该洞察靠另一条好引用幸存、要么整条被排除）。所以：

- **89% < 100% 不是"红线未达"**，是 yield/覆盖效率指标。提升它（streaming 治本 / body 清洗）= 更多洞察幸存 = **覆盖更全**，但不改变"已发布即 100% 可溯源"这一保证。
- 同理校验器三分类 82.1%（<90% 目标）是**保守 over-flag**：安全向（不放坏引用），代价是误杀拉低 yield；不构成红线失守。

> **建议门评确认这一口径**，并据此在 `eval-criteria.md` 标注「可达性 100% = 发布层度量（validator+report-gen 强制）；analyzer 产出 yield 单列为覆盖指标」。本体检暂不改定标文件，留门评裁定。

## 3. 成本现状（provisional · 已补含校验的真实端到端数据）

- **阈值**（`eval-criteria.md` 2026-05-27，Opus-on-relay 重标）：brief ≤¥5 / deep_dive ≤¥15 / initial_digest ≤¥30。
- **analyze-only 锚点**：brief analyze（6 条 arXiv·opus-4-6）≈ ¥0.53；富多源 brief 经 body 上限 **$1.86→$0.99 砍半**（`cost-perf-2026-05-27.md`）。
- **含校验端到端实测**（2026-05-28 双主题重生，analyzer=opus-4-6 / validator=opus-4-7）：
  - swe 一轮（brief+deep_dive 共用一批，30 洞察 / 75 引用）：**$3.63** —— validator **$3.14（86%）** + analyzer $0.49。
  - security 一轮（15 洞察 / 47 引用）：**$1.98** —— analyzer $1.26（拆批重试推高）+ validator $0.72。
  - **关键发现**：**校验是成本大头**（逐条一致性判定跑在最贵的 opus-4-7 上）。含校验后一轮 ≈ ¥14–26，**超**原 analyze-only 重标阈值——原阈值未计校验，且不同主题成本驱动不同（swe=validator 主导、security=analyzer 拆批主导）。
- **处置**：cost 定稿需 ① 按"含校验端到端"重订阈值（口径明确含/不含校验），或 ② validator 降本（Sonnet，待直连 key）。两者均指向 key 类/口径 **附条件**，不阻塞门评（同 DCP-2 处置），但准出② 的"定稿"必须先厘清口径。
